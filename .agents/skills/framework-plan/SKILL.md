# Framework Plan

Help shape a `pi-fiction` idea into a useful plan.

This is a conversation, not a ticket factory. The goal is to understand the creative experience we want to enable before deciding what to build.

Plans live in `.wip/plans/`. The tracker lives at `.wip/README.md`. Completed or abandoned plans can later move to `.wip/plans/.archive/` during cleanup.

## Planning Mindset

Focus on the human and AI creative interaction. Ask what the writer is trying to feel, discover, decide, draft, revise, or continue. Avoid turning the plan into a coding task unless implementation details are necessary to preserve the idea.

A good plan explains:

- what creative problem or opportunity exists
- how the extension should help
- what the user experience should feel like
- what would make the result useful, playful, and not overbearing

## Process

1. **Explore the idea.** Use `$ARGUMENTS` and conversation context as the starting point. If the idea is fuzzy, ask a few useful questions. If it is already clear, proceed.
2. **Find the creative purpose.** Identify the writing moment this plan serves: ideation, character work, worldbuilding, planning, drafting, review, continuity, recovery, momentum, or something else.
3. **Shape the plan.** Draft a concise plan with `Problem`, `Solution`, and optional `Requirements` sections.
4. **Name the plan.** Use `.wip/plans/{topic_slug}-{issue_number}-{name}.md`.
5. **Update the tracker.** Add or move the plan in `.wip/README.md` under its current progress state.
6. **Report the path.** Summarize the plan and ask what should change.

Do not create a plan file if the user only wants to brainstorm. Create or edit files when the user wants the idea captured.

## Naming

Files use:

```text
.wip/plans/{topic_slug}-{issue_number}-{name}.md
```

- `topic_slug`: broad area, lowercase and hyphenated, such as `onboarding`, `prompts`, `characters`, `context`, `review`, `extension`, or `planning`.
- `issue_number`: next two-digit number for that topic, such as `01`, `02`, `03`.
- `name`: short hyphenated summary, usually three to six words.

Example:

```text
.wip/plans/characters-01-discover-pressure-points.md
```

## Progress States

Use simple progress language:

- `spark`: interesting idea, not shaped yet
- `shaping`: actively being discussed or designed
- `ready`: clear enough to build or write
- `active`: currently being implemented
- `done`: finished and ready to archive later
- `parked`: intentionally set aside

## Tracker

`.wip/README.md` is the lightweight workshop board. Keep it useful, not bureaucratic.

Group plans by progress state:

1. Active
2. Ready
3. Shaping
4. Sparks
5. Parked
6. Done

Each entry should be one line:

```markdown
- [plan-id](plans/plan-id.md) - Short human-readable title.
```

When changing progress, update both the plan frontmatter and the tracker in the same edit.

When cleaning up finished work, move done plans to `.wip/plans/.archive/` and update the Done section to point to the archived path or remove it if the user wants a clean board.

## Plan Template

```markdown
---
status: shaping
topic: characters
---

# Discover character pressure points

## Problem

What is missing, confusing, too rigid, or creatively underpowered for the writer or agent?

## Solution

How should `pi-fiction` help? Describe the desired user experience and creative effect.

## Requirements

Optional. Include only constraints that matter for preserving the experience.
```

## Writing Style

Write for future creative collaborators. Be clear, lively, and practical. Avoid implementation minutiae unless it changes the user experience. Never mention that a plan was generated by AI.
